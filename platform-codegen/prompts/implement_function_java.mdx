## Implementation Rules

- The `function_body` replaces the throw statement ONLY — do NOT include the method signature, annotations, or closing brace
- Do NOT modify any other method in the file
- The implementation must be complete — no TODOs, no stubs, no ellipsis
- Handle ALL error cases described in `<strip_description>`
- Use the same patterns visible in other fully-implemented methods in the skeleton (`@Transactional`, exception classes, etc.)
- `function_body` cannot change the file's import section — if your implementation needs a class that's already in `<skeleton_classes>` but not yet imported into `<stub_file_content>` (e.g. a DTO/model/util from another package), or a standard library type like `java.time.LocalDate`, list the required import statement(s) in the `imports` array. Do NOT avoid using the correct class, and do NOT inline a fully-qualified name instead — add it to `imports`. Leave `imports` empty (`[]`) if nothing new is needed. If JPA imports are required, MUST use `jakarta.persistence.*` (NOT `javax.persistence.*`).
- `function_body` also cannot add a new field to the class — if your implementation legitimately needs a repository/service dependency the stub class doesn't already declare, list a plain field declaration in the `fields` array (e.g. `"private ReservationRepository reservationRepository;"`) — no `@Autowired` annotation needed, it's added automatically. Do NOT avoid using a dependency you actually need just because it isn't declared yet. Leave `fields` empty (`[]`) if nothing new is needed.
- **Java Records**: DTOs are Java Records. You MUST use record accessors (e.g., `dto.fieldName()`) and NOT traditional getters (`dto.getFieldName()`).

## Hidden Test Requirements

Use `@SpringBootTest(webEnvironment = RANDOM_PORT)` with `TestRestTemplate`.
Use `@BeforeEach` to reset and seed the database to a known state.
Do NOT use `@Sql` pointing at files under `db/migration/` to reset state — Flyway already runs those on startup and re-running them in a test is invalid. Rely on `@Transactional` for auto-rollback between tests, or seed via the injected repository in `@BeforeEach`.

**MUST FAIL** against the stub (`throw new UnsupportedOperationException("not implemented: ...")`).
**MUST PASS** against the correct gold master implementation.
Test ONLY the method for `<scenario_tag>`.
**Imports** — For the test file, ALWAYS include all necessary imports, such as `java.util.UUID`, `java.util.List`, `java.util.Optional`, `java.time.LocalDate`, etc. If you use a class, you MUST import it.

### Assertion targets (minimums — exceed them)

| Tier   | Min assertions | Min test methods |
|--------|---------------|-----------------|
| EASY   | 60            | 20              |
| MEDIUM | 80            | 25              |
| HARD   | 100           | 30              |

### Required test categories for ALL tiers

1. **Happy path** — correct inputs produce correct HTTP status and response body (5–8 assertions)
2. **Not found** — missing resource returns 404; test with multiple non-existent IDs (3–5 assertions)
3. **Forbidden** — wrong user / insufficient role returns 403 (3–5 assertions)
4. **Validation** — malformed or missing fields return 400 (5–8 assertions)
5. **Boundary values** — zero quantities, empty strings, max lengths, negative numbers (5–8 assertions)
6. **State consistency** — after a mutation, query the repository directly to verify the change (4–6 assertions)
7. **Idempotency** — calling the same endpoint twice returns a consistent result (3–4 assertions)

### Additional categories for MEDIUM

8. **Partial data** — missing optional fields, partial updates behave correctly (3–5 assertions)
9. **Type edge cases** — Unicode strings, very long values, numeric edge cases (3–5 assertions)

### Additional categories for HARD

10. **Concurrency hints** — rapid sequential calls that could race or double-spend (3–5 assertions)
11. **Infrastructure state** — Redis cache miss/hit, circuit breaker states, lock contention (4–6 assertions)
12. **Cascading failures** — upstream unavailable, partial rollback, retry exhaustion (3–5 assertions)
