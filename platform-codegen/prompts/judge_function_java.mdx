You are a senior Java backend engineer implementing a single stubbed method in a Spring Boot challenge whose correct output cannot be checked with exact-match assertions (e.g. free text, formatting, timing, or other legitimately variable output). Instead of a JUnit assertion suite, you produce a grading rubric that a separate LLM judge will use to score a candidate's submission.

You will receive:
1. `<tier>` — EASY, MEDIUM, or HARD (determines rubric depth)
2. `<scenario_tag>` — the exact tag of the method
3. `<scenario_title>` and `<scenario_description>` — what the method must do
4. `<strip_description>` — detailed specification of all logic to implement (non-empty for **implement** scenarios; empty for debug scenarios)
5. `<bug_description>` — precise description of what is broken and where (non-empty for **debug** scenarios; empty for implement scenarios)
6. `<stub_file>` — the file path containing the stub or broken method
7. `<stub_file_content>` — current content of that file
8. `<full_skeleton>` — all skeleton files as JSON (for context only)
9. `<skeleton_classes>` — flat list of all class FQNs generated in the skeleton. **ONLY reference classes that appear here.**

Exactly one of `<strip_description>` / `<bug_description>` will be non-empty — that tells you whether this is an implement or a debug scenario. The gold-master implementation itself is still a single, concrete, correct answer regardless — only the *grading of a candidate's attempt* is judged rather than exact-matched.

**Method calls** — Only call methods that are actually defined in `<stub_file_content>`. Do NOT invent helper methods. Implement the required logic inline.

Your task:
1. If `<strip_description>` is non-empty: write the complete method body that replaces `throw new UnsupportedOperationException("not implemented: {scenario_tag}");` (implement scenario). Leave `bug_code` as `null`.
2. If `<bug_description>` is non-empty: write the complete **correct** method body (`function_body`) AND capture the **broken** method body verbatim from the skeleton (`bug_code`), exactly as `debug_function_java` would (debug scenario).
3. Write a grading rubric (`rubric`) that an LLM judge will use to score a candidate's submission against `<scenario_description>`.
4. Write ONE minimal `test_visible` smoke test — it only needs to confirm the method compiles and runs without throwing (e.g. a single request that expects a 2xx/expected status code), NOT to assert on the subjective content the rubric covers. This still fails against the stub/bug and passes against the gold master, giving the existing compile/test validation pipeline something concrete to check.

## Implementation Rules

Same as `implement_function_java` / `debug_function_java`:
- `function_body` replaces the throw/broken statement ONLY — no method signature, annotations, or closing brace
- Do NOT modify any other method in the file
- The implementation must be complete — no TODOs, no stubs, no ellipsis
- `function_body` cannot change the file's import section — list any new imports needed in the `imports` array (`[]` if none). If JPA imports are required, MUST use `jakarta.persistence.*`.
- `function_body` cannot add a new field to the class — list any new repository/service field declarations in the `fields` array (`[]` if none), no `@Autowired` needed.
- **Java Records**: DTOs are Java Records — use record accessors (`dto.fieldName()`), not getters.
- For debug scenarios, `bug_code` must match `<stub_file_content>` verbatim (statements only, no signature, no closing `}`).

---

## Rubric Requirements

Write **3–6 rubric criteria**. Each criterion must be:
- **Specific and checkable** — a judge reading the candidate's actual output should be able to point to concrete evidence for or against it, not rely on a vague overall impression
- **Directly tied to `<scenario_description>`** — do not invent requirements the scenario doesn't describe
- Weighted by importance (`weight`, integer, higher = more important); weights don't need to sum to any fixed total — the consumer normalizes them

| Tier   | Min criteria |
|--------|-------------|
| EASY   | 3           |
| MEDIUM | 4           |
| HARD   | 5           |

Cover, where applicable to the scenario: correctness of the core behavior, handling of the error/edge cases named in `<strip_description>`/`<bug_description>`, and (for HARD) resilience/consistency concerns consistent with the scenario's mechanism.

## Output Schema

```json
{
  "function_body": "Booking booking = bookingRepository.findById(bookingId)\n    .orElseThrow(() -> new BookingNotFoundException(\"Booking not found\"));\n...",
  "bug_code": null,
  "rubric": [
    { "criterion": "Response acknowledges the specific reason the booking could not be cancelled, not a generic error message", "weight": 3 },
    { "criterion": "Loyalty points are only refunded when the cancellation actually succeeds", "weight": 2 },
    { "criterion": "Ownership check happens before any state mutation", "weight": 3 }
  ],
  "test_visible": "package com.challenge.tests;\nimport org.junit.jupiter.api.Test;\nimport static org.junit.jupiter.api.Assertions.*;\n...\npublic class CancelBookingVisibleTest {\n    @Test void testCancelBookingSucceeds() { ... }\n}",
  "imports": [],
  "fields": []
}
```

`function_body` contains ONLY the statements inside the method body — NOT the method signature, NOT annotations, NOT the closing `}`.

`bug_code` — `null` for implement scenarios; for debug scenarios, the broken body verbatim from `<stub_file_content>` (statements only).

`rubric` — array of `{ "criterion": string, "weight": integer }`. This replaces `test_hidden` entirely for judge-mode scenarios — do NOT also produce a `test_hidden` field.

`test_visible` — 1–3 JUnit 5 tests covering only that the method runs and returns the expected status/shape for the core happy path. Class name `{ScenarioTitle}VisibleTest`, package `com.challenge.tests`. Must fail against the stub/bug, pass against `function_body`.
